## 🧩 Core Libraries & Technologies

EmbPlayer is built on a carefully selected stack of libraries and frameworks that bring its unique host–remote architecture to life.

- **Jetty (Embedded Web Server)** – At the heart of EmbPlayer's remote control functionality is an embedded Jetty web server. It runs locally on the host Android device and serves the web-based control panel. Jetty enables bidirectional communication between the host and the browser-based remote, allowing playback commands, volume adjustments, and stream switching to be transmitted seamlessly over the local network—without needing any cloud services or external routing.

- **QR Code Generation (`io.nayuki.qrcodegen`)** – Used to encode the host's local network URL into a scannable QR code. This eliminates the need for manual IP entry or complex pairing. The library is pure‑Java, lightweight, and works consistently across all Android versions—turning the host screen into an instant connection point.

- **Media Playback Engines (`ijkplayer`, `ExoPlayer`, `VLC`)** – EmbPlayer supports multiple playback engines under the hood, each with its own strengths:
  - **`ijkplayer`** – A lightweight, cross-platform media player based on FFmpeg. It is the default for older Android versions (API 14+) and handles a wide range of formats with solid live‑stream support, offering a great balance between performance and compatibility for legacy devices.
  - **`ExoPlayer`** – Google's modern, customizable media player for Android. It excels at adaptive streaming and is the preferred engine on newer devices (API 16+), delivering superior performance, smooth live playback, and advanced features like dynamic bitrate adaptation. To ensure stability, EmbPlayer includes a custom build of ExoPlayer (located in the `media-X.X.X` folder) with the "different thread triggered rotation exception" disabled, preventing crashes during orientation changes.
  - **`VLC (libVLC)`** – A feature-rich engine based on the popular VLC media player. It provides the broadest codec support and is especially robust for tricky formats and network streams. Available on API 17+ variants.
  
  The app automatically selects the most suitable engine for each media source and Android version, ensuring reliable playback—including live streams—across the broadest possible device range.

- **OkHttp and SSL Management** – EmbPlayer uses **OkHttp** as its HTTP client for loading websites, fetching streams, and performing network operations. To ensure maximum compatibility across Android versions (from 4.0 to the latest), a dedicated SSL management layer is implemented:
  - **Conscrypt** – A modern TLS provider that backports TLS 1.2 and 1.3 to older Android devices, enabling secure connections even on legacy systems.
  - **Insecure SSL Fallback** – When secure connections fail (e.g., due to outdated certificates or network restrictions), the app can temporarily fall back to trusting all certificates, ensuring the user can still access content.
  - **Legacy TLS Support** – For Android 4.2 and below, the app uses a custom socket factory that enables older TLS versions (TLSv1, TLSv1.1, TLSv1.2) to work around platform limitations.
  - **Automatic Loader Switching** – The `SiteLoader` class transparently swaps between two internal loader implementations when a request fails, automatically retrying with different SSL configurations or switching to HTTP, guaranteeing that the content is loaded with the highest possible success rate.
  All these mechanisms are centrally controlled by `SslConfigManager`, which dynamically adjusts settings based on the device's Android version, making EmbPlayer resilient to a wide variety of network environments and server configurations.

- **NewPipeExtractor** – Used to extract video and playlist information from YouTube links. This open‑source library parses YouTube's public data without requiring an API key, allowing EmbPlayer to fetch stream URLs, metadata, and playlist contents directly from standard YouTube links (videos, live streams, and playlists). It works entirely offline after the initial fetch and respects YouTube's terms of service through pure client‑side extraction, keeping the app independent of external services.

- **Skytube Fork of NewPipeExtractor** – For Android versions below API 21 (Android 5.0), EmbPlayer uses a compatibility fork of NewPipeExtractor from the Skytube project. This variant maintains the same extraction capabilities while supporting older Android runtimes (API 14–20), ensuring feature parity across all supported devices.

- **RxJava – The Reactive Foundation** – RxJava is the backbone of EmbPlayer's concurrency model. It allows the app to write asynchronous, event-driven code in a clean and composable way. Instead of manually juggling threads, callbacks, and state, RxJava provides a rich set of operators and schedulers that simplify parallel execution, error handling, and data streaming. This is crucial for EmbPlayer because the host must handle multiple tasks simultaneously: serving the web UI, responding to remote commands, playing media, and updating metadata – all without blocking the user interface.

  - **ForkJoinPool – The Parallel Engine** – At the heart of EmbPlayer's CPU‑bound task execution lies the `ForkJoinPool`, a specialized thread pool designed specifically for parallel computation. Unlike traditional thread pools that use a single shared queue, `ForkJoinPool` implements a **work‑stealing algorithm**: every worker thread maintains its own deque (double‑ended queue) of tasks. When a thread finishes its own work, it can "steal" pending tasks from the *back* of another thread's queue. This design minimizes contention between threads and keeps all CPU cores productively busy, dramatically accelerating operations like media parsing, stream validation, metadata extraction, and QR code generation. EmbPlayer customises this pool further by adjusting thread priorities and execution order – some operations benefit from a last‑in‑first‑out (LIFO) strategy (which improves cache locality and reduces overhead for recursively split tasks), while others use the opposite order to balance fairness and prevent starvation. The pool automatically scales to the device's available processor cores, ensuring optimal performance on everything from low‑end quad‑core phones to high‑end octa‑core tablets. For legacy Android devices that lack native `ForkJoinPool` support, EmbPlayer transparently substitutes a repackaged implementation, keeping these performance benefits available across the entire supported device range.

  - **`DisposableTools` – Smart Threading & Task Execution** – Building directly on RxJava and the `ForkJoinPool` engine, `DisposableTools` acts as the central nervous system for all threading operations. It extends RxJava by defining a set of specialised schedulers that match the specific workloads of the app. For instance, one scheduler leverages the work‑stealing `ForkJoinPool` for CPU‑intensive operations, another is optimised for blocking I/O (network, disk) and can grow its thread count as needed, while additional schedulers are fine‑tuned for media playback and web server startup with different thread priorities – ensuring the most critical tasks receive CPU time first. The layer also provides a uniform way to submit any unit of work, automatically capturing and logging errors, and includes built‑in timeout mechanisms to prevent hanging operations. The entire system is designed to be compatible with Android versions as old as 4.0 by gracefully falling back to alternative thread‑pool implementations when the standard `ForkJoinPool` is unavailable.

  - **`Recyclable` – Automatic Task Lifecycle & Cleanup** – While `DisposableTools` manages *how* and *where* tasks run, `Recyclable` handles *tracking* and *disposing* of them over time. In a dynamic app like EmbPlayer, many background tasks are spawned and finished constantly – generating QR codes, checking stream availability, refreshing channel lists, serving web requests, and more. Without proper cleanup, these tasks can accumulate, causing memory leaks and wasted CPU cycles. `Recyclable` solves this by providing a self‑managing container that holds active tasks. Each task, when created, is assigned a unique slot; when the task completes (or fails), it automatically removes itself from the container, freeing its slot for future reuse – this prevents the container from growing indefinitely and avoids manual bookkeeping. The container also offers a bulk‑cleanup operation: when the host activity or service is destroyed, it can dispose of every remaining task in one go, ensuring absolutely no background work survives longer than needed. This pattern is especially valuable in EmbPlayer's remote‑control scenario, where the user may switch streams, reconnect browsers, or close the app at any moment, and the system must remain responsive and memory‑lean.

- **Android Version-Specific Support Libraries** – To maintain compatibility across nearly a decade of Android releases, EmbPlayer incorporates a range of support libraries and compatibility shims, carefully selected to ensure smooth operation on every supported Android version.

Together, these libraries enable EmbPlayer to deliver a smooth, responsive, and universally compatible experience—from ancient tablets running Android 4.0 to the latest smartphones, with full live‑streaming capability in every engine.
